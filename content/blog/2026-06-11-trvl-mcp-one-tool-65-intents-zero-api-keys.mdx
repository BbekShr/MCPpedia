## One Tool to Route Them All

Most travel MCP servers drown you in endpoints. [Trvl Mcp](/s/trvl-mcp) takes the opposite approach — one smart tool, 65 ways to use it, and zero API keys standing between you and real travel data.

That's not a limitation. That's a design philosophy.

<SectionLabel>What It Does</SectionLabel>

The entire server is built around a single tool: `travel`. You pass it an `intent` — a string that can be one of 65 aliases — and it routes your request to the right data source automatically.

**Flights? Hotels? Trains? Ferries? Lounges?** It's all handled by the same interface. Under the hood, `trvl-mcp` queries Google Flights, Google Hotels, Trivago, Airbnb, Booking.com, Hostelworld, Ferryhopper, and European ground transport networks — all without you ever touching an API key.

<Callout type="fire">No API keys required. Free, API-first access to Google Flights, Google Hotels, Trivago, Airbnb, Booking.com, and more — out of the box.</Callout>

The intent aliases cover an impressively wide surface area:

- **Air travel**: flights, price alerts, award sweet spots, baggage rules, lounges
- **Accommodation**: hotels, Airbnb, hostels via Hostelworld, Booking.com properties
- **Ground transport**: rental cars, trains, buses, ferries across European networks
- **Trip intelligence**: destination intel, weather, and more

<PullQuote>65 aliases, one routing layer — this is what a well-abstracted travel agent looks like.</PullQuote>

The `award_sweet_spots` intent alone is worth the install for frequent flyers. Finding where your miles stretch furthest is notoriously painful — having an AI agent that can surface that data on demand, mid-conversation, is genuinely useful.

<SectionLabel>The Design Bet</SectionLabel>

There's a real architectural opinion embedded in this server. Most MCP tools give you granular, separate tools for each action — one for flights, one for hotels, one for weather. That's orthodox MCP design.

`trvl-mcp` bets that **intent routing** is actually better for AI agents. Instead of the model deciding which tool to call, it just describes what it wants and the server figures out the rest. 65 aliases act as compatibility shims for natural language variations — `flights`, `flight`, `find_flights`, `search_flights` all resolve to the same behavior.

<PullQuote>This is opinionated software. The 65 aliases aren't bloat — they're a compatibility layer for the messy, unpredictable way AI models actually phrase requests.</PullQuote>

That bet has tradeoffs. You get less granular control per-intent, and debugging a routing failure is harder when there's only one tool to inspect. But for the use case — an AI travel agent that just *works* — the simplicity wins.
